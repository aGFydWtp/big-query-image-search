データ登録と検索のフロー

テキスト → 画像 セマンティック検索システム（GCP / BigQuery ネイティブ）

取込はバッチ（純 SQL）、検索はリクエスト時に BigQuery で実行。常駐の埋め込みサービスを持たない。

データ登録（バッチ取込） 検索（Cloud Run API）

単一ベクトル空間

取込・検索の両方で同じ gemini-embedding-2-preview・同じ 3072 次元を使い、画像とテキストを同じ空間に埋め込む。これが検索精度の前提。

BigQuery に集約

埋め込み生成（AI.GENERATE_EMBEDDING）も近傍探索（VECTOR_SEARCH）も BigQuery 内で実行。専用サービスが不要。

共有契約テーブル

image_embeddings（3072 次元 / COSINE）が取込と検索をつなぐ唯一の接点。スキーマ変更は両者の再検証トリガー。

埋め込みを生成して差分投入（取込の心臓部）

Object Table の画像に AI.GENERATE_EMBEDDING を適用してベクトル化し、MERGE … WHEN NOT MATCHED で未登録の URI だけを投入。再実行しても重複しない（冪等）。成功行・3072 次元・画像 MIME のみ通す。

-- generate_embeddings.sql（要点） MERGE image_embeddings AS tgt USING ( SELECT src.uri AS image_uri, src.embedding, src.content_type FROM AI.GENERATE_EMBEDDING( MODEL gemini_embedding_model, TABLE image_object_table, STRUCT(3072 AS output_dimensionality)) WHERE src.status = '' -- 成功行のみ AND ARRAY_LENGTH(src.embedding) = 3072 AND src.content_type LIKE 'image/%' ) AS src ON tgt.image_uri = src.image_uri WHEN NOT MATCHED THEN INSERT (...); -- 既存 URI は再投入しない

VECTOR INDEX 作成（近傍探索の高速化）

embedding 列に IVF・COSINE のベクトルインデックスを張る。データ量が約 10MB 未満の間は未 populate で、その時はブルートフォース探索にフォールバック。新規行は BigQuery がマネージドに非同期更新（手動再構築不要）。

sql/vector_index.sql image_embeddings_idx index_type=IVF / distance_type=COSINE
冪等性がカギ： ②③④⑥ は OR REPLACE / IF NOT EXISTS、⑤ は MERGE による差分投入。画像を追加したら ①（投入）→ ⑤（埋め込み生成）を回すだけで、既存分には触れず新規分だけが検索対象に加わる。

検索検索フロー（Cloud Run API）

自然言語クエリを POST /search に送ると、Cloud Run 上の Go サービスがまず（任意で）Vertex AI Gemini で英語リライトを生成し、続いて BigQuery で「2 系統のクエリ埋め込み生成 → VECTOR_SEARCH ×2 → RRF 融合」を1 つのジョブとして実行し、類似画像を返す。検索 UI（SPA）は同サービスに同梱され同一オリジンで配信されるため CORS 設定は不要。

英語リライト生成（任意・rrf_vec チャネル）

query_en 未指定かつ REWRITE_ENABLED=true（既定）のとき、Vertex AI Gemini（gemini-2.5-flash）で日本語クエリを英語の画像説明に変換し、RRF の 2 系統目（rrf_vec チャネル）に用いる。失敗・タイムアウトは非致命：warn ログのみ記録し、英語リライトなし（生クエリのみ）の単一チャネル検索にフォールバックする（5xx にしない）。REWRITE_ENABLED=false では Noop となり Vertex AI を呼ばない。

3

検索 SQL を 1 ジョブで実行（2 系統 + RRF 融合）

生クエリ @query と英語リライト @query_en を AI.GENERATE_EMBEDDING で各 3072 次元にベクトル化し（取込と同一モデル・同一次元）、VECTOR_SEARCH を 2 系統ぶん実行。各系統の順位を RRF（k=60）で融合し rrf_score 降順に並べて top_k 件返す。@query_en が空なら rewrite 行は生成されず raw 単一チャネルに縮約（埋め込み・探索とも 1 回）。各系統の候補数は @candidate_k = max(top_k, 50)。識別子は config 注入、値はパラメータバインド。

-- internal/search/templates/search.sql（要点） WITH query_inputs AS ( SELECT 'raw' AS source, @query AS content UNION ALL SELECT 'rewrite', @query_en FROM UNNEST([1]) WHERE NULLIF(@query_en, '') IS NOT NULL), -- 空なら単一チャネル query_embeddings AS ( SELECT source, embedding FROM AI.GENERATE_EMBEDDING( MODEL gemini_embedding_model, TABLE query_inputs, STRUCT(3072 AS output_dimensionality)) WHERE status = ''), vector_matches AS ( SELECT query.source, base.image_uri, base.content_type, distance, ROW_NUMBER() OVER (PARTITION BY query.source ORDER BY distance) AS rank FROM VECTOR_SEARCH(TABLE image_embeddings, 'embedding', TABLE query_embeddings, top_k => @candidate_k, distance_type => 'COSINE')), rrf AS ( SELECT image_uri, ANY_VALUE(content_type) AS content_type, MIN(distance) AS distance, SUM(1.0 / (60 + rank)) AS rrf_score FROM vector_matches GROUP BY image_uri) SELECT image_uri, content_type, distance FROM rrf ORDER BY rrf_score DESC, distance ASC LIMIT @top_k;

登録と検索をつなぐ「共有契約」

  • 同一モデル：取込の埋め込み生成も、検索のクエリ埋め込みも gemini_embedding_model（gemini-embedding-2-preview）
  • 同一次元：両者とも output_dimensionality = 3072
  • 同一距離タイプ：VECTOR INDEX も VECTOR_SEARCH も COSINE
  • 接点テーブル：image_embeddings（image_uri / embedding / content_type / generated_at）

この 4 点が一致しているからこそ、テキストクエリと画像が同じベクトル空間で比較でき、意味的に近い画像が返る。どれか 1 つでもずれると検索結果が壊れるため、スキーマ変更は取込・検索の両 spec の再検証トリガーになる。
